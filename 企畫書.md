# 企業模塊化組件系統企畫書

## Context

BillW 主要業務為接案開發企業管理系統（ERP 類），客戶需求功能大同小異（打卡、財務、請假等）。為縮短交付週期，計劃預先開發一套可複用的模塊化組件系統。

模組化是在**程式碼層級**做拆分（Maven 子模組 / 前端 feature 資料夾），部署時仍然是標準的 4 個 Docker 容器（frontend、backend、postgres、redis）。客戶不需要的功能透過設定檔停用（Feature Toggle），而非拆分容器。

## 一、技術棧總覽

### 核心技術（每個專案必備）

| 層級 | 技術選型 | 說明 |
|------|---------|------|
| 後端框架 | Spring Boot 3.x (Java 21) | 單一 JAR 部署，內含所有模組 |
| ORM | JPA / Hibernate | 資料存取層 |
| 排程 | Quartz Scheduler | 定時任務 |
| 前端 Web | React + TypeScript + Vite + MUI + Zustand | 單一前端 build |
| 前端 App | Flutter (Dart) | 行動端應用 |
| 資料庫 | PostgreSQL | 單一資料庫，模組各自的表以前綴區分 |
| 快取 | Redis | Session、快取 |
| 容器化 | Docker + Docker Compose | 4 個容器部署 |
| 反向代理 | Nginx | 前後端路由 |
| DB 遷移 | Flyway | 版本化遷移腳本 |
| CI/CD | GitHub Actions / GitLab CI | 自動化建構與部署 |

### 選配技術（依客戶需求加入）

| 技術 | 何時加入 | 說明 |
|------|---------|------|
| MinIO | 需要大量檔案存儲時 | S3 相容物件存儲（+1 容器） |
| Elasticsearch | 需要全文搜尋或大量日誌時 | 搜尋引擎（+1 容器） |
| RabbitMQ | 系統規模大、需非同步處理時 | 訊息佇列（+1 容器） |
| Prometheus + Grafana | 需要監控時 | 指標收集與可視化（+2 容器） |

### 模組啟停機制

透過 Spring Boot 的 `application.yml` 設定檔控制模組開關：

```yaml
modules:
  auth: true           # 必啟
  organization: true   # 必啟
  attendance: true
  leave: true
  workflow: true
  finance: false       # 此客戶不需要
  payroll: false
  # ...
```

每個模組使用 `@ConditionalOnProperty` 控制是否載入，關閉的模組不會啟動 Controller、Service、排程等。

### 模組間通訊

因為所有模組在同一個 JVM 內，模組間直接透過 **Spring 的依賴注入**呼叫 Service，不需要 REST 或訊息佇列：

```java
// 在 module-leave 中直接注入 module-organization 的 Service
@Service
public class LeaveServiceImpl implements LeaveService {
    private final EmployeeService employeeService; // 來自 module-organization
    // ...
}
```

需要解耦的場景（如「請假核准後通知」）使用 **Spring Event**（ApplicationEventPublisher），同 JVM 內的事件機制，零額外基礎設施。

### 資料庫策略

- 單一 PostgreSQL 資料庫
- 模組的表以前綴區分：`auth_users`, `org_departments`, `att_records`, `leave_requests` 等
- 所有表必備欄位：`id` (UUID PK), `created_at`, `updated_at`, `deleted_at`（軟刪除）
- Flyway 遷移腳本按模組分目錄：`db/migration/auth/`, `db/migration/organization/` 等

|------|------|------|
| 傳輸層 | HTTPS 強制 | Nginx SSL Termination |
| 認證層 | JWT 驗證、Token 過期 | Spring Security |
| 授權層 | RBAC + 資料權限 | @PreAuthorize + 自訂 Filter |
| 資料層 | PII 欄位加密、ORM 防注入 | Jasypt + JPA |
| 密碼層 | BCrypt/Argon2 加鹽雜湊 | PasswordEncoder |
| 日誌層 | 敏感資料脫敏 | AOP |
| 環境層 | 環境變數注入 | .env + Docker |
| API 層 | Rate Limit、CORS、CSRF | Spring Security Filter |

## 六、模組間依賴關係

```
module-common ← 所有模組依賴
    ↓
module-auth ← 所有業務模組依賴
    ↓
module-organization ← 大多數業務模組依賴
    ↓
module-workflow ← 請假、財務等審批類依賴
    ↓
┌──────────────┬──────────────┬──────────────┐
attendance      leave          finance        notification ← 所有模組可推送
    ↓              ↓              ↓
    └──────────────┴──────────────┘
                   ↓
               module-payroll（整合考勤+請假+財務）
```

模組間透過 Spring 依賴注入 + Spring Event 溝通，無額外基礎設施需求。

## 十、風險與注意事項

1. **單體擴展性**：若未來單一客戶流量極大，可考慮拆分為微服務，但目前單體完全足夠
2. **module-common 變更**：影響所有模組，需謹慎修改並確保向下相容
3. **模組間耦合**：雖在同一 JVM，仍應保持模組間僅透過 Service 介面 + Event 通訊，避免直接存取對方 Repository
4. **Feature Toggle**：關閉的模組 Flyway 遷移仍會執行（表結構存在但無資料），確保不影響效能
5. **檔案存儲**：小規模用本地磁碟 + Docker Volume，大規模再加 MinIO 容器
