# 企業模塊化組件系統 — 開發與測試整合規劃書

---

## 一、開發階段總覽與模塊依賴圖

### 模塊依賴關係

```
module-common ← 所有模組依賴（基礎設施層）
    ↓
module-auth ← 所有業務模組依賴（認證授權層）
    ↓
module-organization ← 大多數業務模組依賴（組織資料層）
    ↓
module-workflow ← 請假、財務等審批類模組依賴（流程引擎層）
    ↓
┌───────────────┬───────────────┬───────────────┐
module-attendance  module-leave    module-finance   module-notification
    ↓                 ↓               ↓                  ↑（監聽所有事件）
    └─────────────────┴───────────────┘
                      ↓
                 module-payroll（整合考勤+請假+財務資料）
```

### 四階段開發路線圖

| 階段 | 時程 | 模塊 | 目標 |
|------|------|------|------|
| 第一階段：核心基礎 | ~6-8 週 | common → Docker 基礎設施 → auth → organization → workflow → 前端共用框架 | 系統能登入、管理組織、發起審批 |
| 第二階段：核心業務 | ~8-10 週 | notification → attendance → leave → system → audit → finance | 日常營運功能完備 |
| 第三階段：擴展業務 | ~8-10 週 | payroll → project → document → report → crm | 進階管理與分析 |
| 第四階段：進階功能 | ~6-8 週 | inventory → meeting → announcement | 附加功能模塊 |

## 二、Sprint 規劃原則

- **Sprint 週期**：2 週為一個 Sprint
- **Sprint 目標**：每個 Sprint 交付 **1 個完整模塊**（後端 + 前端 + 測試通過）
- **較大模塊**（如 workflow、finance）可拆為 2 個 Sprint
- **每個 Sprint 結構**：
  - Day 1-2：後端 Entity + Repository + Flyway 遷移
  - Day 3-5：後端 Service + Controller + Event
  - Day 6-7：後端單元測試 + 整合測試
  - Day 8-9：前端頁面開發 + API 串接
  - Day 10：前後端聯調 + 模塊驗收測試

## 三、Git 分支策略

```
main ─────────────────────────────────────────── 正式發布
  └── develop ────────────────────────────────── 開發主線
        ├── feature/module-common ────────────── 模塊開發分支
        ├── feature/module-auth
        ├── feature/module-organization
        └── fix/auth-token-refresh ───────────── Bug 修復分支
```

- 每個模塊建立 `feature/module-[名稱]` 分支
- 模塊完成並通過所有測試後，PR 合併至 `develop`
- 階段結束時，`develop` 合併至 `main` 產出 Release
- 跨模塊依賴的 PR：先合併被依賴的模塊，再開發依賴方

#### Sprint 1-1：module-common（共用基礎）— 2 週

**前置條件**：無（這是第一個模塊）

**後端開發步驟**：

1. **建立 Maven 父 POM**
   - 建立 `backend/pom.xml`，定義 Spring Boot 3.x parent、Java 21、全局依賴版本管理
   - 定義 `<modules>` 區塊，先加入 `module-common` 和 `app`

2. **建立 module-common 子模塊**
   - `backend/module-common/pom.xml`：引入 Spring Boot Starter Web、JPA、Security、Redis、Validation、Lombok
   - 建立以下套件結構：
     ```
     com.enterprise.common/
     ├── config/        → RedisConfig, CorsConfig, SecurityConfig
     ├── dto/           → ApiResponse<T>, PageResponse<T>, ErrorResponse
     ├── entity/        → BaseEntity (id UUID, createdAt, updatedAt, deletedAt)
     ├── enums/         → Status(ACTIVE/INACTIVE), EventType
     ├── event/         → BaseEvent (eventId, timestamp, source)
     ├── exception/     → GlobalExceptionHandler, BusinessException, ResourceNotFoundException
     ├── security/      → JwtTokenProvider, JwtAuthenticationFilter, SecurityUtils
     ├── util/          → DateUtil, MoneyUtil, EncryptUtil, StringUtil
     ├── audit/         → @Auditable AOP 切面定義
     ├── annotation/    → @RequirePermission, @Auditable, @ModuleEnabled
     └── feature/       → @ConditionalOnModule 自訂條件注解
     ```

3. **逐一實作各套件**
   - `BaseEntity`：使用 `@MappedSuperclass`，id 用 `@GeneratedValue(strategy = GenerationType.UUID)`，`deletedAt` 配合 `@SQLDelete` 和 `@Where` 實作軟刪除
   - `ApiResponse<T>`：統一回應格式 `{ code, message, data, timestamp }`
   - `GlobalExceptionHandler`：使用 `@RestControllerAdvice` 攔截所有例外，轉為統一格式
   - `JwtTokenProvider`：產生 Access Token (15min) + Refresh Token (7days)，使用 HMAC-SHA256
   - `JwtAuthenticationFilter`：繼承 `OncePerRequestFilter`，從 Header 解析 JWT
   - `@ConditionalOnModule`：包裝 `@ConditionalOnProperty(prefix = "modules", name = "xxx", havingValue = "true")`

4. **建立 app 啟動模塊**
   - `backend/app/pom.xml`：引入 `module-common`（後續逐步加入其他模塊）
   - `application.yml`：設定 DataSource (PostgreSQL)、Redis、JWT secret、模塊開關區塊
   - `Application.java`：`@SpringBootApplication(scanBasePackages = "com.enterprise")`

**基礎設施開發步驟**：

5. **Docker 配置**
   - `docker/local/docker-compose.yml`：定義 frontend、backend、postgres、redis 四個容器
   - `docker/local/Dockerfile.backend`：Multi-stage build（Maven build → JRE runtime）
   - `docker/local/Dockerfile.frontend`：Node build → Nginx 靜態檔案
   - `docker/local/nginx.conf`：前端靜態 + `/api` 反向代理至 backend:8080
   - `env/local/.env`：DB_NAME, DB_USER, DB_PASSWORD, REDIS_PORT, JWT_SECRET 等
   - `env/.env.example`：所有變數名稱（不含真實值）

**前端開發步驟**：

6. **建立 React 專案骨架**
   - `npx create-vite frontend-web --template react-ts`
   - 安裝依賴：`@mui/material`, `zustand`, `axios`, `react-router-dom`, `react-i18next`
   - 建立共用框架：
     ```
     frontend-web/src/shared/
     ├── api/          → axiosInstance（攔截器：自動帶 JWT、401 跳轉登入、統一錯誤處理）
     ├── auth/         → useAuth hook, TokenManager (存取/刷新 Token)
     ├── components/   → DataTable, FormBuilder, ConfirmDialog, LoadingOverlay
     ├── hooks/        → usePagination, useDebounce, usePermission
     ├── i18n/         → zh-TW.json, en.json 翻譯檔
     ├── store/        → authStore (Zustand: user, token, permissions)
     ├── theme/        → MUI 主題定制 (色系、字型、間距)
     ├── types/        → ApiResponse<T>, PageResponse<T>, BaseEntity
     └── utils/        → dateFormat, moneyFormat, validators
     ```
   - 建立路由框架：`App.tsx` 含 ProtectedRoute（驗證 JWT）+ Feature Toggle 條件路由
   - 建立 Layout：側邊選單 + 頂部導航列 + 內容區域

**測試（module-common 完成時）**：

| 測試類型 | 測試項目 | 預期結果 |
|---------|---------|---------|
| 單元測試 | `JwtTokenProvider.generateToken()` / `validateToken()` / `parseToken()` | Token 產生、解析、過期驗證正確 |
| 單元測試 | `ApiResponse.success()` / `ApiResponse.error()` | 回應格式符合 `{ code, message, data, timestamp }` |
| 單元測試 | `GlobalExceptionHandler` 各種例外處理 | 不同例外對應正確 HTTP Status 與錯誤格式 |
| 單元測試 | `DateUtil`, `MoneyUtil`, `EncryptUtil` 各方法 | 轉換、計算、加解密正確 |
| 整合測試 | `docker-compose up` 四容器啟動 | 所有容器 healthy，backend 可連接 PostgreSQL 和 Redis |
| 整合測試 | Spring Boot 啟動（無業務模塊） | Application context 正常載入，無 Bean 錯誤 |
| 前端測試 | `axiosInstance` 攔截器 | 自動帶 Token、401 導向登入 |
| 前端測試 | `DataTable` 組件渲染 | 正確顯示表頭、分頁、排序 |

#### Sprint 1-2：module-auth（認證與授權）— 2 週

**前置條件**：module-common 已完成並合併至 develop

**後端開發步驟**：

1. **建立模塊目錄**
   ```
   backend/module-auth/
   ├── pom.xml                → 依賴 module-common
   └── src/main/java/com/enterprise/auth/
       ├── controller/        → AuthController, UserController, RoleController, PermissionController
       ├── service/           → AuthService, UserService, RoleService, PermissionService (interface + impl)
       ├── repository/        → UserRepository, RoleRepository, PermissionRepository, LoginLogRepository
       ├── entity/            → User, Role, Permission, RolePermission, UserRole, LoginLog, MfaConfig
       ├── dto/               → LoginRequest, LoginResponse, RegisterRequest, UserDTO, RoleDTO
       ├── config/            → AuthModuleConfig (@ConditionalOnProperty)
       └── event/             → UserCreatedEvent, UserDisabledEvent
   ```

2. **定義 Entity**
   - `User`：extends BaseEntity，欄位 username, passwordHash, email, phone, status, failedAttempts, lockedUntil, mfaEnabled
   - `Role`：extends BaseEntity，欄位 name, code, description
   - `Permission`：extends BaseEntity，欄位 name, code, type(MENU/API/DATA), resource, action
   - `RolePermission`：role_id + permission_id 多對多關聯表
   - `UserRole`：user_id + role_id 多對多關聯表
   - `LoginLog`：user_id, ip, device, loginTime, location, success(boolean)
   - `MfaConfig`：user_id, type(TOTP), secret, enabled

3. **撰寫 Flyway 遷移腳本**
   - `db/migration/auth/V1__create_auth_tables.sql`：
     ```sql
     CREATE TABLE auth_users (
       id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
       username VARCHAR(50) NOT NULL UNIQUE,
       password_hash VARCHAR(255) NOT NULL,
       email VARCHAR(100),
       phone VARCHAR(20),
       status VARCHAR(20) DEFAULT 'ACTIVE',
       failed_attempts INT DEFAULT 0,
       locked_until TIMESTAMP,
       mfa_enabled BOOLEAN DEFAULT FALSE,
       created_at TIMESTAMP DEFAULT NOW(),
       updated_at TIMESTAMP DEFAULT NOW(),
       deleted_at TIMESTAMP
     );
     -- auth_roles, auth_permissions, auth_role_permissions, auth_user_roles, auth_login_logs, auth_mfa_configs 同理
     -- 建立索引：auth_users(username), auth_users(email), auth_login_logs(user_id, login_time)
     ```

4. **實作 Service 層**
   - `AuthServiceImpl`：
     - `login(LoginRequest)`：驗證帳密 → 檢查鎖定 → 產生 JWT → 記錄登入日誌 → 返回 LoginResponse
     - `refreshToken(String refreshToken)`：驗證 Refresh Token → 產生新 Access Token
     - `logout(String userId)`：將 Token 加入 Redis 黑名單
   - `UserServiceImpl`：
     - `createUser(RegisterRequest)`：密碼 BCrypt 加密 → 存入 DB → 發佈 `UserCreatedEvent`
     - `getUserById(UUID)`、`listUsers(Pageable, filters)`、`updateUser()`、`disableUser()`
     - `checkLoginAttempts()`：5 次失敗鎖定 30 分鐘
   - `RoleServiceImpl`：CRUD + 指派權限
   - `PermissionServiceImpl`：CRUD + 樹狀選單權限查詢

5. **實作 Controller 層**
   - `POST /api/v1/auth/login`：登入
   - `POST /api/v1/auth/refresh`：刷新 Token
   - `POST /api/v1/auth/logout`：登出
   - `GET/POST/PUT/DELETE /api/v1/users`：使用者 CRUD
   - `GET/POST/PUT/DELETE /api/v1/roles`：角色 CRUD
   - `GET /api/v1/permissions`：權限列表
   - `POST /api/v1/roles/{roleId}/permissions`：指派權限

6. **模塊開關配置**
   - `AuthModuleConfig`：`@Configuration @ConditionalOnProperty(prefix = "modules", name = "auth", havingValue = "true", matchIfMissing = true)`
   - auth 為必啟模塊，`matchIfMissing = true`

**前端開發步驟**：

7. **建立 auth feature**
   ```
   frontend-web/src/features/auth/
   ├── pages/          → LoginPage.tsx, UserListPage.tsx, UserFormPage.tsx, RoleListPage.tsx, RoleFormPage.tsx, PermissionPage.tsx
   ├── components/     → LoginForm.tsx, UserTable.tsx, RolePermissionTree.tsx, MfaSetupDialog.tsx
   ├── api/            → authApi.ts (login, refresh, logout), userApi.ts (CRUD), roleApi.ts (CRUD)
   ├── hooks/          → useLogin.ts, useUsers.ts, useRoles.ts
   ├── types/          → User.ts, Role.ts, Permission.ts, LoginRequest.ts, LoginResponse.ts
   └── index.ts
   ```
8. **LoginPage**：帳號 + 密碼 + 員工編號 → 呼叫 `/api/v1/auth/login` → 存 Token 至 authStore → 跳轉首頁
9. **UserListPage**：DataTable 顯示使用者列表，支援搜尋、分頁、新增/編輯/停用操作
10. **RoleListPage**：角色列表 + 點擊角色展開權限樹（勾選指派）

**測試（module-auth 完成時）**：

| 測試類型 | 測試項目 | 詳細說明 |
|---------|---------|---------|
| 單元測試 | `AuthServiceImpl.login()` | Mock UserRepository → 正確帳密回傳 Token；錯誤密碼拋出 AuthException；帳號鎖定拋出 LockedException |
| 單元測試 | `AuthServiceImpl.login()` 鎖定邏輯 | 連續 5 次失敗 → failedAttempts=5, lockedUntil 為 30 分鐘後 |
| 單元測試 | `UserServiceImpl.createUser()` | 驗證密碼經過 BCrypt 加密、UserCreatedEvent 被發佈 |
| 單元測試 | `RoleServiceImpl.assignPermissions()` | 角色正確關聯權限 |
| 整合測試 | Flyway 遷移 `V1__create_auth_tables.sql` | Testcontainers PostgreSQL 容器中執行遷移，表結構正確建立 |
| 整合測試 | `UserRepository` CRUD | 真實 DB 執行新增、查詢、更新、軟刪除 |
| 整合測試 | JWT + Redis 登出 | 登出後 Token 存入 Redis 黑名單，再次使用該 Token 應返回 401 |
| 整合測試 | 完整登入流程 | 從 API 層發起 login → 取得 Token → 用 Token 呼叫受保護 API → 成功 |
| API 測試 | `POST /api/v1/auth/login` | 正確帳密 → 200 + Token；錯誤帳密 → 401；缺少欄位 → 400 |
| API 測試 | `GET /api/v1/users` 無 Token | → 401 Unauthorized |
| API 測試 | `GET /api/v1/users` 有 Token 無權限 | → 403 Forbidden |
| API 測試 | `GET /api/v1/users` 正確角色 | → 200 + 分頁資料 |
| API 測試 | `POST /api/v1/users` 重複 username | → 409 Conflict |
| 前端測試 | `LoginForm` 組件 | 輸入帳密 → 點擊登入 → 呼叫 API → 成功跳轉；空欄位顯示驗證錯誤 |
| 前端測試 | `useAuth` hook | Token 存取、刷新、登出邏輯正確 |
| 模塊開關測試 | `modules.auth=true` | auth 相關 Bean 全部載入 |

   - `GET/POST/PUT/DELETE /api/v1/departments`
   - `GET/POST/PUT/DELETE /api/v1/positions`

   ```
       ├── service/      → WorkflowDefinitionService, WorkflowEngineService, WorkflowTaskService

       ├── event/        → ApprovalCompletedEvent, ApprovalRejectedEvent, ApprovalPendingEvent
       └── config/       → WorkflowModuleConfig
   ```

2. **定義 Entity**
   - `WorkflowDefinition`：name, code, category(LEAVE/EXPENSE/PURCHASE), version, status(DRAFT/PUBLISHED), description
   - `WorkflowNode`：definitionId, type(START/APPROVAL/CONDITION/END), name, approverType(ROLE/POSITION/DEPARTMENT_MANAGER/SPECIFIC_USER), approverId, approvalStrategy(ANY/ALL), sortOrder, timeoutHours
   - `WorkflowCondition`：sourceNodeId, targetNodeId, field, operator(GT/LT/EQ/BETWEEN), value（如金額 > 10000 走高階主管審批）
   - `WorkflowInstance`：definitionId, businessType, businessId, initiatorId, status(PENDING/APPROVED/REJECTED/CANCELLED), currentNodeId
   - `WorkflowTask`：instanceId, nodeId, assigneeId, status(PENDING/APPROVED/REJECTED/FORWARDED), comment, operatedAt
   - `WorkflowHistory`：instanceId, nodeId, operatorId, action, comment, operatedAt

   - 索引重點：`wf_tasks(assignee_id, status)`, `wf_instances(initiator_id, status)`

4. **實作 WorkflowEngine（核心）**
   - `startWorkflow(businessType, businessId, initiatorId)`：
     1. 根據 businessType 查找對應的 WorkflowDefinition
     2. 建立 WorkflowInstance
     3. 找到第一個 APPROVAL 節點
     4. 根據 approverType 解析審批人（呼叫 EmployeeService 取得部門主管等）
     5. 建立 WorkflowTask
     6. 發佈 `ApprovalPendingEvent`
   - `approve(taskId, comment)`：
     1. 更新 Task 狀態為 APPROVED
     2. 檢查是否會簽全部完成（ALL 策略）
     3. 找到下一個節點（評估條件分支）
     4. 如果是 END 節點 → 更新 Instance 為 APPROVED → 發佈 `ApprovalCompletedEvent`
     5. 如果是下一個 APPROVAL 節點 → 建立新 Task → 發佈 `ApprovalPendingEvent`
   - `reject(taskId, comment)`：更新 Task + Instance → 發佈 `ApprovalRejectedEvent`
   - `forward(taskId, newAssigneeId)`：轉簽
   - `getMyPendingTasks(userId, pageable)`：我的待辦
   - `getInstanceProgress(instanceId)`：流程進度追蹤

5. **實作 Controller 層**
   - `GET/POST/PUT /api/v1/workflow/definitions`：流程定義 CRUD
   - `POST /api/v1/workflow/definitions/{id}/publish`：發布流程
   - `POST /api/v1/workflow/start`：發起審批
   - `POST /api/v1/workflow/tasks/{taskId}/approve`：同意
   - `POST /api/v1/workflow/tasks/{taskId}/reject`：拒絕

## 五、每個模塊完成時的標準測試流程

無論哪個模塊，完成開發後都必須依照以下順序執行測試：

### Step 1：後端單元測試

```bash
cd backend
mvn test -pl module-[模塊名] -am
```

- **工具**：JUnit 5 + Mockito
- **範圍**：Service 層所有公開方法
- **覆蓋率要求**：
  - Service 業務邏輯：≥ 80%
  - 金額/稅務計算邏輯：100%
  - Controller 層：不需單元測試（由 API 測試覆蓋）
  - Entity/DTO：不需測試
- **必測項目**：
  - 正向流程（Happy Path）
  - 邊界條件（空值、極大值、極小值）
  - 異常路徑（不存在的資源、權限不足、餘額不足等）
  - Event 發佈驗證（`verify(eventPublisher).publishEvent(any(XxxEvent.class))`）

### Step 2：後端整合測試

```bash
cd backend
mvn verify -pl module-[模塊名] -am -P integration-test
```

- **工具**：Spring Boot Test + Testcontainers (PostgreSQL + Redis)
- **測試內容**：
  1. **Flyway 遷移測試**：該模塊 + 所有前置模塊的遷移腳本在空資料庫執行無錯誤
  2. **Repository CRUD**：真實 DB 環境下的新增/查詢/更新/軟刪除
  3. **跨模塊 Service 呼叫**：驗證依賴注入的其他模塊 Service 可正常呼叫
  4. **Spring Event 事件鏈**：發佈事件 → 監聽方正確接收並處理
  5. **模塊啟停測試**：將此模塊設為 `false` → 應用程式仍正常啟動、不報 Bean 錯誤

### Step 3：API 測試

```bash
cd backend
mvn test -pl app -P api-test
```

- **工具**：REST Assured 或 Spring MockMvc
- **每個 endpoint 必測**：

| 測試項目 | 預期 |
|---------|------|
| 無 Authorization Header | 401 |
| Token 過期 | 401 |
| 無權限的角色 | 403 |
| 正確請求 | 200/201 + 正確回應格式 |
| 缺少必填欄位 | 400 + 詳細錯誤 |
| 資源不存在 | 404 |
| 重複資料（如重複 username） | 409 |
| 分頁參數 | page, size, sort 正常運作 |
| 多條件篩選 | 各篩選條件組合正確 |

### Step 4：前端測試

```bash
cd frontend-web
npx vitest run --coverage
```

- **工具**：Vitest + React Testing Library
- **必測項目**：
  1. 頁面組件渲染（是否正確顯示）
  2. 表單驗證（必填欄位、格式驗證）
  3. API 呼叫 Mock（提交表單 → 呼叫正確 API → 成功/失敗回饋）
  4. 使用者互動（點擊、篩選、分頁）

### Step 5：前後端聯調測試

- 啟動完整 Docker 環境
- 手動操作每個頁面的核心流程
- 驗證 API 回傳格式在前端正確解析顯示
- 驗證錯誤處理（後端返回 4xx/5xx → 前端顯示友善錯誤訊息）

### Step 6：回歸測試

```bash
cd backend
mvn verify -P all-tests
```

- **執行範圍**：所有已完成模塊的測試套件
- **目的**：確保新模塊沒有破壞舊模塊的功能
- **關注點**：
  - module-common 的變更是否向下相容
  - 新增的 Spring Event Listener 是否影響既有事件流
  - 新 Flyway 遷移腳本是否與舊腳本衝突
  - 共用的 Redis 快取 Key 是否衝突

### Step 7：模塊驗收清單（Definition of Done）

每個模塊完成後，必須逐項確認：

- [ ] 模塊可透過 Feature Toggle 關閉，系統正常啟動

|---------|------|
| **模塊隔離** | 是否直接存取了其他模塊的 Repository？（禁止！只能透過 Service 介面） |
| **Event 解耦** | 弱依賴場景是否使用 Spring Event 而非直接呼叫？ |
| **模塊開關** | 是否有 `@ConditionalOnProperty` 配置？關閉時是否會導致 Bean 錯誤？ |
| **金額類型** | 金額欄位是否使用 BigDecimal？是否有 double/float？ |
| **統一回應格式** | 所有 API 是否返回 ApiResponse<T>？ |
| **安全** | 敏感 API 是否有 @PreAuthorize？密碼是否 BCrypt？日誌是否脫敏？ |
| **命名規範** | DB 表/欄位 snake_case？Java camelCase？前綴是否正確？ |
| **雙語註釋** | 是否包含繁體中文 + English 雙語？ |
| **軟刪除** | 是否使用 deletedAt 而非實體刪除？ |
| **索引** | FK 和常用查詢欄位是否建立索引？ |

```yaml
      - name: Run Integration Tests

      - name: Check Coverage
```

- `main` 分支合併後自動部署至測試環境

## 八、異常隔離與系統穩定性

因為所有模塊在同一 JVM 中運行，需要防止單一模塊錯誤拖垮整個系統：

| 隔離機制 | 實作方式 |
|---------|---------|
| **線程池隔離** | 各模塊的非同步任務使用獨立的 `@Async` 線程池（`TaskExecutor`），防止某模塊的大量任務佔滿公用線程池 |
| **超時控制** | 所有跨模塊 Service 呼叫加上 `@Transactional(timeout = 30)`，避免長時間鎖表 |
| **記憶體監控** | 使用 Spring Boot Actuator 暴露 `/actuator/health`、`/actuator/metrics`，監控各模塊的記憶體使用 |
| **熔斷降級** | 非核心模塊（如 notification、report）的調用加上 try-catch，失敗不影響主流程（如請假審批通過但通知發送失敗 → 請假仍然生效） |
| **Flyway 隔離** | 各模塊遷移腳本按目錄分離，單一模塊遷移失敗不阻塞其他模塊 |
| **排程隔離** | 各模塊的 Quartz Job 使用不同的 Job Group，單一 Job 異常不影響其他排程 |

### 9.1 Flyway 版本號規範（必讀，避免啟動錯誤）

**問題**：Flyway 的版本號是**全局唯一**的，若每個模塊都使用 `V1__...`，多模塊合併後會報 `Found more than one migration with version` 錯誤。

**正確做法**：所有遷移腳本統一放在 `module-common/src/main/resources/db/migration/` 單一路徑下，以模塊編號作為版本號前綴：

```
db/migration/
├── V1101__auth_create_tables.sql     # module-auth
├── V1102__auth_seed_default_roles.sql
├── V2101__org_create_tables.sql      # module-organization
├── V3101__workflow_create_tables.sql # module-workflow
├── V4101__noti_create_tables.sql     # module-notification
├── V5101__att_create_tables.sql      # module-attendance
├── V6101__leave_create_tables.sql    # module-leave
├── V7101__sys_create_tables.sql      # module-system
├── V8101__audit_create_tables.sql    # module-audit
├── V9101__fin_create_tables.sql      # module-finance
├── V10101__pay_create_tables.sql     # module-payroll
├── V11101__proj_create_tables.sql    # module-project
├── V12101__doc_create_tables.sql     # module-document
├── V13101__report_create_tables.sql  # module-report
├── V14101__crm_create_tables.sql     # module-crm
├── V15101__inv_create_tables.sql     # module-inventory
├── V16101__meeting_create_tables.sql # module-meeting
├── V17101__ann_create_tables.sql     # module-announcement
└── V181xx 起                         # 未來新增模塊
```

**application.yml 只需一個路徑**（不需每加模塊就修改）：

```yaml
spring:
  flyway:
    locations: classpath:db/migration
    enabled: true
```

### 9.2 Optional 模塊依賴注入模式（必讀，避免啟動失敗）

**問題**：若模塊 A 直接強制注入模塊 B 的 Service，當 B 被 Feature Toggle 關閉時，Spring 啟動報 `NoSuchBeanDefinitionException`。

**正確模式**：對所有跨模塊依賴使用 `@Autowired(required = false)` 加降級處理：

```java
@Service
public class LeaveRequestServiceImpl implements LeaveRequestService {

    // ========================================
    // Optional 依賴：workflow 模塊可能未啟用 / Optional dependency: workflow module may be disabled
    // ========================================
    @Autowired(required = false)
    private WorkflowEngineService workflowEngineService;

    @Override
    public LeaveRequestDTO submitRequest(LeaveRequestDTO dto) {
        // ... 驗證、建立 LeaveRequest ...

        if (workflowEngineService != null) {
            // workflow 已啟用：走正式審批流程
            workflowEngineService.startWorkflow("LEAVE", request.getId(), dto.getEmployeeId());
        } else {
            // workflow 未啟用：降級為自動核准
            request.setStatus(LeaveStatus.AUTO_APPROVED);
            log.warn("Workflow module disabled, leave request auto-approved: {}", request.getId());
        }
        return leaveMapper.toDTO(leaveRequestRepository.save(request));
    }
}
```

**需要 Optional 注入的模塊清單**：

| 注入方 | Optional Service | 降級行為 |
|-------|-----------------|---------|
| module-leave | WorkflowEngineService | 自動核准，記錄 warn log |
| module-finance (費用報銷) | WorkflowEngineService | 同上 |
| module-attendance (補卡) | WorkflowEngineService | 同上 |
| module-payroll | AttendanceService | 跳過考勤相關扣款，薪資計為底薪 |
| module-payroll | LeaveService | 跳過無薪假扣款 |
| module-meeting | NotificationService | 不發送與會者通知 |
| module-announcement | NotificationService | 不發送推播通知 |

### 9.3 Hibernate 6.x 語法修正（Spring Boot 3.x）

| 舊語法（Spring Boot 2.x） | 新語法（Spring Boot 3.x） |
|--------------------------|--------------------------|
| `@Where(clause = "deleted_at IS NULL")` | `@SQLRestriction("deleted_at IS NULL")` |
| `@Type(type = "json")` | `@JdbcTypeCode(SqlTypes.JSON)` |
| `@SQLDelete(...)` | 語法不變，仍可使用 |

**BaseEntity 正確寫法（Spring Boot 3.x）**：

```java
@MappedSuperclass
@SQLRestriction("deleted_at IS NULL")   // 注意：不是 @Where
@SQLDelete(sql = "UPDATE #{#tableName} SET deleted_at = NOW() WHERE id = ?")
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;

    @CreatedDate
    @Column(name = "created_at", updatable = false)
    private LocalDateTime createdAt;

    @LastModifiedDate
    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    @Column(name = "deleted_at")
    private LocalDateTime deletedAt;
}
```

### 9.4 新增模塊時需更新的兩個 POM（常被遺漏）

每次新增模塊，必須同時更新**兩個**地方：

**1. backend/pom.xml（父 POM）**：
```xml
<modules>
    <module>module-common</module>
    <!-- 其他已有模塊 -->
    <module>module-xyz</module>   <!-- 新增 -->
    <module>app</module>
</modules>
```

**2. backend/app/pom.xml**：
```xml
<dependencies>
    <!-- 其他已有模塊 -->
    <dependency>
        <groupId>com.enterprise</groupId>
        <artifactId>module-xyz</artifactId>   <!-- 新增 -->
        <version>${project.version}</version>
    </dependency>
</dependencies>
```

> 父 POM 的 `<modules>` 控制「哪些子模塊參與建構」，app 的 `<dependencies>` 控制「哪些類別被打包進 JAR」。兩者都要更新。

### 9.5 前端 Feature Toggle 路由實作範例

**從後端取得啟用模塊清單並動態渲染路由**：

```typescript
// shared/store/featureStore.ts
export const useFeatureStore = create<FeatureState>((set) => ({
  enabledModules: {},
  fetchFeatures: async () => {
    const res = await systemApi.getFeatures(); // GET /api/v1/system/features
    set({ enabledModules: res.data });
  },
}));
```

```tsx
// App.tsx 路由設定
function AppRoutes() {
  const { enabledModules } = useFeatureStore();
  return (
    <Routes>
      <Route element={<ProtectedRoute />}>
        <Route element={<Layout />}>
          <Route path="dashboard" element={<DashboardPage />} />
          {enabledModules.attendance && <Route path="attendance/*" element={<AttendanceRoutes />} />}
          {enabledModules.leave && <Route path="leave/*" element={<LeaveRoutes />} />}
          {enabledModules.finance && <Route path="finance/*" element={<FinanceRoutes />} />}
        </Route>
      </Route>
    </Routes>
  );
}
```

> 側邊選單的項目也需依 `enabledModules` 做條件過濾，確保關閉的模塊不出現在選單中。

### 9.6 module-organization 員工編號暫代方案

module-system 的 `SequenceRuleService` 到第二階段才開發，但 module-organization 在第一階段就需要員工編號。暫時在 organization 模塊內實作簡易序號邏輯：

```java
// EmployeeRepository.java
@Query(value = "SELECT COALESCE(MAX(CAST(SUBSTRING(employee_no, 10) AS INTEGER)), 0) " +
               "FROM org_employees WHERE employee_no LIKE :prefix%",
       nativeQuery = true)
Integer findMaxSequenceByPrefix(@Param("prefix") String prefix);

// EmployeeServiceImpl.java
private synchronized String generateEmployeeNo() {
    String prefix = "EMP-" + Year.now().getValue() + "-";
    int max = Optional.ofNullable(employeeRepository.findMaxSequenceByPrefix(prefix)).orElse(0);
    return prefix + String.format("%04d", max + 1);
}
```

等 module-system 完成後，再將此方法改為呼叫 `SequenceRuleService.getNextSequence("EMP")`。

## 十、模塊依賴矩陣（部署至客戶專案時的組合參考）

| 目標功能 | 必須啟用 | 可選推薦 |
|---------|---------|---------|
| 基本系統（任何客戶） | `common`, `auth` | `system`, `audit`, `notification` |
| 員工管理 | + `organization` | — |
| 打卡考勤 | + `organization` | + `workflow`（補卡審批）|
| 請假管理 | + `organization`, **`workflow`** | + `notification` |
| 審批流程 | + `organization`, `workflow` | + `notification` |
| 財務管理 | + `auth`, **`workflow`** | + `notification` |
| 薪資管理 | + `organization`, **`attendance`**, **`leave`**, **`finance`** | + `notification` |
| 專案管理 | + `organization` | + `notification` |
| 文件管理 | + `auth` | + `workflow`（審批）|
| 報表分析 | + `auth` | + 視資料來源加入對應模塊 |
| 客戶管理 | + `auth` | + `organization`, `notification` |
| 庫存管理 | + `auth` | + `notification`（低庫存警示）|
| 會議管理 | + `organization` | **`notification`**（與會通知）|
| 公告系統 | + `auth` | + `organization`（發布範圍）, `notification` |

**強依賴說明（粗體 = 不啟用需實作降級邏輯）**：
- `module-leave` → **`module-workflow`**：請假走審批，若 workflow 未啟用降級為自動核准
- `module-payroll` → **`module-attendance` + `module-leave`**：薪資整合考勤/請假資料，缺少時對應項目計 0
- `module-finance`（費用報銷）→ **`module-workflow`**：同請假降級邏輯

**不可關閉的模塊**（`matchIfMissing = true`）：
```yaml
modules:
  auth: true        # 認證必啟
  organization: true  # 幾乎所有業務模塊依賴，建議固定開啟
```
